Chapter 01

Why veth Exists

Modern Linux networking is fundamentally built around isolation. As cloud infrastructure and containerization have become the norm, processes increasingly need to operate in hermetically sealed networking environments — without the overhead of a full virtual machine.

Processes may require isolated routing tables, isolated interfaces, isolated firewall rules, and even isolated localhost environments, all while sharing the same physical host.

Linux Solves This With

  • Network namespaces
  • Virtual Ethernet devices (veth)
  • Linux bridges
  • iptables / nftables

Powers Everything In

  • Docker & Podman
  • Kubernetes pods
  • CTF sandboxes
  • Service meshes

A veth pair behaves exactly like a virtual cable — a bidirectional pipe through the kernel. Anything entering one side immediately exits the other:

veth pair — virtual cable
[  veth0  ] <═══════════════> [  veth1  ]
     ↑                               ↑
  Host side                   Namespace side
💡 Key Insight The entire veth pair exists inside the kernel — no physical hardware, no NIC required. Packets traverse it at memory-copy speed.
Chapter 02

Mental Model

The best way to understand virtual Ethernet is to think physically. The analogy maps almost perfectly to real hardware.

Physical Ethernet
Computer A
    │
  cable
    │
Computer B
Virtual Ethernet (veth)
Namespace A
    │
veth pair
    │
Namespace B

The key difference is that both ends exist entirely inside the kernel — there is no wire, no NIC, no physical signal. The kernel handles the "transmission" by directly moving packet buffers between two net_device structures in memory.

This makes veth pairs:

⚡ Extremely Fast

  • No serialization
  • No DMA transfers
  • No IRQs

🔒 Fully Isolated

  • Separate ARP
  • Separate routing
  • Separate firewall

🔬 Perfectly Observable

  • tcpdump works
  • ss works
  • ip tools work
Chapter 03

What Is a Network Namespace?

A network namespace is a complete, isolated networking stack — a full software replica of the networking environment that the kernel provides, but hermetically separated from every other namespace on the system.

Think of it as giving a process its own private version of the entire Linux network subsystem.

Component Description Isolated?
interfacesNetwork devices (eth0, lo, etc.)✓ Yes
routing tableKernel IP routing decisions✓ Yes
ARP tableIP → MAC address mappings✓ Yes
iptablesPacket filtering rules✓ Yes
127.0.0.1Loopback / localhost✓ Yes
socketsBound ports and connections✓ Yes
📦 Practical Reality Processes inside different namespaces behave exactly like separate physical machines, even though they share the same CPU, RAM, and disk. Each one has its own 127.0.0.1, its own port space, its own routing decisions.
Chapter 04

Your First Namespace

Creating a network namespace is a single command. The kernel allocates a new, isolated networking stack and gives it a name you can reference later.

bash
# Create a named network namespace
sudo ip netns add ns1

# Verify it was created
ip netns list

Behind the scenes, the kernel creates a reference file at:

filesystem
/var/run/netns/ns1

This file is a bind-mount that holds a reference to the kernel namespace object, keeping it alive even if no processes are running inside it.

💡 Tip You can execute commands inside a namespace using ip netns exec <name> <command>. This is how you interact with the isolated environment from the host.
Chapter 05

Creating a veth Pair

A veth pair is always created as two connected interfaces simultaneously. You cannot create one end without the other — they are born as a matched pair, like two ends of a cable.

bash
# Create a veth pair: veth0 ↔ veth1
sudo ip link add veth0 type veth peer name veth1

# Verify both interfaces now exist
ip link
Result — both interfaces in the host namespace
HOST NAMESPACE
┌──────────────────────────────────────┐
│   veth0  <═══════════════>  veth1   │
│   (DOWN)                    (DOWN)   │
└──────────────────────────────────────┘

At this point, both interfaces are in the default namespace and are in the DOWN state. The next step is to move one end into a namespace — effectively "plugging in" one end of the cable.

Chapter 06

Moving an Interface Into a Namespace

Moving one end of the veth pair into a namespace creates the virtual cable between the host and the namespace. Once moved, veth1 disappears from the host's interface list and becomes visible only inside ns1.

bash
# Move veth1 into namespace ns1
sudo ip link set veth1 netns ns1

# Verify: veth1 is gone from host
ip link

# Verify: veth1 appears inside ns1
sudo ip netns exec ns1 ip link
After moving veth1 into ns1
HOST NAMESPACE          NAMESPACE ns1
┌───────────┐           ┌───────────┐
│   veth0   │<═════════>│   veth1   │
│   (DOWN)  │           │   (DOWN)  │
└───────────┘           └───────────┘
      ↑                       ↑
  Stays here           Lives here now
⚠️ Important An interface can only exist in one namespace at a time. Moving it is a one-way operation per invocation — the interface is relocated, not copied.
Chapter 07

Assigning IP Addresses

With the virtual cable in place, we need to assign IP addresses to each end so that the kernel can route packets between them.

bash — host side
# Assign IP to veth0 on the host
sudo ip addr add 192.168.100.1/24 dev veth0
bash — namespace side
# Assign IP to veth1 inside ns1
sudo ip netns exec ns1 ip addr add 192.168.100.2/24 dev veth1
InterfaceLocationIP AddressSubnet
veth0Host namespace192.168.100.1/24
veth1Namespace ns1192.168.100.2/24

Both addresses must be on the same subnet (/24 here, meaning 192.168.100.0–255) so the kernel's routing logic considers them directly reachable without needing a gateway.

Chapter 08

Bringing Interfaces UP

Linux interfaces are created in the DOWN state by default. You must explicitly bring each one up. This applies to both ends of the veth pair, and also to the loopback interface inside the namespace — a commonly forgotten step that causes mysterious failures.

  1. Bring up veth0 on the host
    bash
    sudo ip link set veth0 up
  2. Bring up veth1 inside ns1
    bash
    sudo ip netns exec ns1 ip link set veth1 up
  3. Bring up loopback inside ns1

    Without this, applications inside the namespace that bind to 127.0.0.1 will fail silently.

    bash
    sudo ip netns exec ns1 ip link set lo up
Chapter 09

Testing Connectivity

With both interfaces up and IPs assigned, you now have a fully functioning virtual network. Test it by pinging the namespace's IP from the host:

bash
# Ping the namespace from the host
ping 192.168.100.2
output — success
PING 192.168.100.2 (192.168.100.2) 56(84) bytes of data.
64 bytes from 192.168.100.2: icmp_seq=1 ttl=64 time=0.082 ms
64 bytes from 192.168.100.2: icmp_seq=2 ttl=64 time=0.071 ms
64 bytes from 192.168.100.2: icmp_seq=3 ttl=64 time=0.068 ms
💡 Notice the latency Sub-millisecond RTT. This is what "kernel-internal" means — no physical wire, no NIC interrupt. The packet traverses only kernel memory.

You can also ping in the other direction — from inside the namespace back to the host:

bash
sudo ip netns exec ns1 ping 192.168.100.1
Chapter 10

Packet Flow Internals

Understanding exactly what happens inside the kernel when a ping traverses a veth pair demystifies a lot of container networking. Here is the precise sequence:

  1. Routing Lookup

    The kernel consults its routing table to determine which interface can reach 192.168.100.2.

    bash
    ip route
# → 192.168.100.0/24 dev veth0 proto kernel scope link src 192.168.100.1
  2. ARP Resolution

    The kernel sends an ARP broadcast: "Who owns 192.168.100.2?" The namespace's networking stack replies with the MAC address of veth1. The kernel caches this mapping.

  3. Ethernet Frame Transmission

    The ICMP packet is wrapped in an Ethernet frame addressed to veth1's MAC and passed to veth0's transmit path. The veth driver's dev_queue_xmit() redirects the frame directly into veth1's receive queue. No hardware involved.

  4. ICMP Handling & Reply

    The namespace's networking stack processes the ICMP echo request, constructs an echo reply, and sends it back through the same path in reverse.

Chapter 11

Running Programs Inside Namespaces

You can execute any program inside a network namespace using ip netns exec. The program sees only the interfaces and routing table of that namespace, as if it were running on an isolated machine.

bash — open an interactive shell
sudo ip netns exec ns1 bash

Once inside, standard networking commands see only the namespace's interfaces:

bash — inside ns1
ip addr
# Output:
# 1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536 ...
#     inet 127.0.0.1/8 scope host lo
# 2: veth1: <BROADCAST,MULTICAST,UP,LOWER_UP> ...
#     inet 192.168.100.2/24 scope global veth1
💡 Use Case This is exactly how container runtimes like Docker work — they call clone(2) with CLONE_NEWNET, move a veth peer into the new namespace, then execute the container process. The container's eth0 is that veth peer.
Chapter 12

Python Sockets Across veth

veth pairs work transparently with all socket-based applications. Here's a minimal TCP client-server example spanning the host and namespace:

python — server (run on host)
import socket

s = socket.socket()
s.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
s.bind(("192.168.100.1", 31337))
s.listen(1)

print("[+] Waiting for connection...")
conn, addr = s.accept()
print(f"[+] Connected from {addr}")

while True:
    data = conn.recv(1024)
    if not data: break
    print(f"[host] received: {data.decode()}")
    conn.send(b"ACK: " + data)
python — client (run inside ns1)
# sudo ip netns exec ns1 python3
import socket

s = socket.socket()
s.connect(("192.168.100.1", 31337))

s.send(b"hello from ns1")
reply = s.recv(1024)
print(f"[ns1] got: {reply.decode()}")
s.close()
🔬 What this demonstrates The kernel routes packets through the veth pair using normal socket paths. From the application's perspective, this is indistinguishable from a real network connection to a remote machine.
Chapter 13

Multiple Namespaces

A single veth pair connects exactly two endpoints. To connect multiple namespaces, you need a bridge — a software Layer 2 switch that can forward frames between many veth interfaces simultaneously.

Multi-namespace topology
   ns1           ns2           ns3
    │              │              │
  veth1          veth3          veth5
    │              │              │
  veth0          veth2          veth4
    └──────────────┴──────────────┘
                   │
               [ br0 bridge ]
                   │
             Host network

This is exactly the architecture Docker uses for its default bridge network mode. Each container gets a veth pair; one end lives in the container's namespace, the other is attached to the docker0 bridge.

Chapter 14

Linux Bridges

A Linux bridge is a software Layer 2 switch implemented entirely in the kernel. It learns MAC addresses, forwards frames, and supports spanning tree — just like a physical managed switch.

bash — create and configure a bridge
# Create the bridge device
sudo ip link add br0 type bridge
sudo ip link set br0 up

# Assign an IP to the bridge (becomes gateway)
sudo ip addr add 10.0.0.1/24 dev br0

# Attach veth host-end to the bridge
sudo ip link set veth0 master br0
sudo ip link set veth2 master br0  # second namespace

# Inspect bridge state
bridge link show
💡 Slaves vs Masters When you set veth0 master br0, veth0 becomes a "port" on the bridge. The bridge now owns Layer 2 forwarding for all attached interfaces. The bridge's IP becomes the gateway for all namespaces attached to it.
Chapter 15

Container Networking Architecture

Every time Docker starts a container, it executes a sequence of kernel operations that you now fully understand:

Docker bridge networking — full stack
Container Namespace (e.g., myapp)
┌──────────────────────────────┐
│  eth0  ←  inside container  │
└────────┬─────────────────────┘
         │  veth pair
         │
┌────────┴─────────────────────┐
│  vethXXXXX  (host side)      │
│       │                      │
│  [ docker0 bridge ]          │  ← 172.17.0.1
│       │                      │
│  host network stack          │
│       │                      │
│  iptables NAT (MASQUERADE)   │  ← outbound internet
└──────────────────────────────┘

Kubernetes extends this foundation significantly, but the primitive never changes:

Docker adds

  • Bridge docker0
  • iptables NAT rules
  • Port mapping (DNAT)
  • DNS via embedded server

Kubernetes adds

  • VXLAN overlay tunnels
  • CNI plugin chain
  • kube-proxy iptables
  • eBPF datapath (Cilium)

Chapter 17

Common Failure Modes

The same handful of mistakes account for the vast majority of veth/namespace failures. Know these patterns and you'll resolve issues in seconds.

Interface DOWN

Most common. Interfaces are created DOWN by default. Both ends must be UP.

fix
ip link set veth0 up
ip netns exec ns1 \
  ip link set veth1 up

Missing Loopback

Namespace lo is DOWN by default. Apps binding to 127.0.0.1 silently fail.

fix
ip netns exec ns1 \
  ip link set lo up

Subnet Mismatch

Both endpoints must share the same subnet or you need an explicit route entry.

bad config
veth0: 192.168.1.1/24
veth1: 192.168.2.1/24
# Different subnets — no route!

Missing Route

Especially in multi-namespace setups. Verify routes exist on both sides.

diagnose
ip route
ip netns exec ns1 ip route
Chapter 18

Systems-Level Understanding

At the kernel level, a veth pair is implemented as two net_device structures that hold a pointer to each other. When the kernel needs to "transmit" a packet out one side, it calls the veth driver's transmit function, which directly injects the packet buffer into the peer's receive queue.

Kernel-level packet path
Application calls send()
        │
        ▼
  Socket layer (TCP/UDP)
        │
        ▼
  IP layer (routing, NAT, netfilter)
        │
        ▼
  veth0->ndo_start_xmit()
        │   ← this is just a function call
        ▼
  veth1's receive queue
        │
        ▼
  netif_rx() → softirq
        │
        ▼
  IP layer (in ns1)
        │
        ▼
  Application recv()

The critical insight: the "transmission" is just a function call that moves a pointer. There is no serialization, no interrupt, no DMA transfer. This is why veth pairs achieve near-zero latency and why container networking can be so efficient.

📖 Further Reading The veth driver source is at drivers/net/veth.c in the Linux kernel. The key function is veth_xmit(). It's a remarkably short file for something so foundational.
